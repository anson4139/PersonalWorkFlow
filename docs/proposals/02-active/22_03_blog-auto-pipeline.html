三、背景與目標

3.1 現況痛點

Blog 已具備完整的 AI 生文、封面圖、插圖功能，但全流程仍需人工觸發：手動輸入 URL → 手動按生文 → 手動按生封面 → 手動按生插圖。每天若要持續發文，人工成本高且容易中斷。

3.2 目標

  • Phase 1（每日）：每天早上 07:00 自動從 2 個 RSS 來源（VentureBeat + Ars Technica） 加上 FinMind TaiwanStockNews 全量掃描 收集新聞，無需人工觸發。
  • 自動篩選並對應到 6 個分類（AI / 科技 / 資安 / 產業 / 國際 / 財經金融） 3 個分類（AI / 產業 / 資安），每類最多發佈一篇。
  • 每分類逐篇嘗試 ai-generate（選拔）：失敗的草稿保留入 D1（供 Phase 2 分析）；第一篇通過品質門檻者繼續 ai-cover → ai-illustrate → 發佈（status = published）。
  • 不影響現有後台操作方式，管理員仍可手動使用 AiGeneratePage 生成任意文章。
  • Phase 2（每週）：每週一自動讀取 D1 posts（含草稿與已發佈），依 14 家目標企業分組疊加 title + excerpt，以 GPT-5.5 生成產業趨勢洞察，持久化入 insights 表，形成可回溯的知識庫。
  • 前端 /trends 頁面以時間軸呈現 insights，可按公司 / 影響層級 / 時間範圍篩選。
  • 部署初期一次性回填（Phase 0）：Phase 1 剛上線時 D1 posts 幾乎是空的，若此時直接啟動 Phase 2 Cron 將得到 0 筆分析素材。因此在 Phase 2 正式啟動前，需執行一次性本機腳本 scripts/backfill-insights.mjs --weeks=4，繞過 D1 posts、改呼叫 FinMind TaiwanStockNews 歷史 API，取得過去 N 週 14 家企業的真實新聞標題，每家每週取前 20 筆送 GPT-5.5 生成趨勢洞察，以 data_source='FINMIND' 寫入 D1 insights 表。此回填完成後，/trends 頁面上線當天即有歷史縱深資料可顯示；後續每週 Cron 改以 data_source='D1_POSTS' 持續累積。此作業為一次性手動步驟，不重複執行，費用約 $0.15（~56 次 GPT-5.5 呼叫）。

3.3 設計原則

  • 最小侵入：現有三個 AI endpoint 不改對外行為，pipeline 只新增一條呼叫路徑。
  • Auth 隔離：Pipeline 使用獨立 PIPELINE_SECRET 認證，不與管理員 session 混用。
  • 失敗不阻塞：單篇失敗直接跳過，不重試，保證 Cron 在 15 分鐘內完成。
  • 備選不斷供：每分類預選前 3 5 名，主選失敗依序遞補。

Phase 1 全程在單一 Cron Worker 內依序執行，預計耗時 ~14 分鐘 ~7 分鐘，在 Cloudflare Unbound 模式 15 分鐘上限內。Phase 2 週報每家企業有 500ms 間隔（GPT rate-limit），預計 14 家 × 2s ≈ 30 秒。

4.2 為何選 Cron Worker 而非多個 Cron

每篇耗時約 2.3 分鐘（生文 ~25s + 封面 ~50s + 插圖 ~50s），6 篇合計約 14 分鐘 3 篇合計約 7 分鐘。單一 Cron 在 Unbound 模式下可完整執行，不需要 KV 協調多個 Cron 的執行序，實作最簡單且最可靠。

4.3 Auth 設計：不影響現有後台

現有 requireAdmin 只接受管理員 session cookie。Pipeline Cron Worker 無法持有 session，因此新增一個獨立的 fallback 認證路徑：

// _auth.ts 修改（唯一需要動的現有程式）
export async function requireAdmin(request, env) {
  // 原有 session 驗證邏輯不變
  const sessionOk = await checkSession(request, env);
  if (sessionOk) return null;

  // 新增：Pipeline Secret fallback（僅供 Cron Worker 使用）
  const authHeader = request.headers.get("Authorization") ?? "";
  if (authHeader === `Bearer ${env.PIPELINE_SECRET}`) return null;

  return json({ error: "Unauthorized" }, 401);
}
隔離保證： PIPELINE_SECRET 只存在 Cloudflare 環境變數中，不出現在任何程式碼或 git history。管理員後台的 session 認證路徑完全不受影響。

4.4 翻譯策略：讓現有系統處理

所有英文來源（VentureBeat、Ars Technica）不需要額外翻譯步驟。ai-generate 接收 URL 後透過 r.jina.ai 抓取全文，再由 GPT 直接輸出繁體中文，翻譯已內建在現有流程中。Pipeline 只需把文章 URL 傳入即可。

5.3 評分邏輯與選篇機制

每個分類的所有候選文章依以下規則計分，取前 3 名作為備選佇列（主選 + 備選 1 + 備選 2） 5 名作為備選佇列（主選 + 備選 1 ~ 4）：

因子條件分數
新鮮度（高）發布時間距今 ≤ 6 小時+3
新鮮度（低）發布時間距今 ≤ 24 小時+1
來源深度（高）VentureBeat+2
來源深度（中）Ars Technica / FinMind+1
摘要豐富度RSS description 長度 ≥ 100 字元+1
關鍵字命中title 含分類核心關鍵字+1
品牌加分 ★ NEW title 含：NVIDIA/輝達、AMD、Intel、Microsoft/微軟、Anthropic、OpenAI、SpaceX、Google、TSMC/台積電、Amazon、OpenClaw、Broadcom +2

同分時優先取 pubDate 較新者。分數計算在記憶體內完成，不需額外 AI 呼叫。

5.4 兩層去重策略

Layer 1：Hard Dedup（跨天）

  • 儲存：Cloudflare KV，namespace PIPELINE_DEDUP
  • Key：url:{sha256(article_url)}
  • TTL：30 天自動清除
  • 時機：每篇文章送入 ai-generate 前先查 KV；成功發佈後寫入 KV
  • 效果：同一篇文章 30 天內不會被重複發佈

Layer 2：Soft Dedup（批次內）

  • 儲存：當次 Cron 記憶體，不持久化
  • 方法：同分類候選佇列中，比對 title 關鍵字重疊率；同批次已選文章的 title tokens 集合中，若新文章重疊超過 60% 則排除
  • 效果：同一天不會有主題高度重複的兩篇文章

七、風險與防呆機制

7.1 系統風險與處理方式

#風險描述防呆機制
R1 Cron 重複觸發（Cloudflare 偶發重試） 每日執行鎖：Cron 啟動時先寫入 KV lock:YYYYMMDD；若 key 已存在則立即終止。
R2 OpenAI 費用失控 Phase 1：每次 Cron 硬限 3 篇（每分類 1 篇）。Phase 2：週報每家企業 1 次 GPT-5.5 呼叫，最多 14 次/週，費用可控。
R3 生文內容空白或垃圾 Sanity check：title 非空 + content_html ≥ 500 字元；否則視為失敗並自動嘗試備選。
R4 D1 寫入部分成功（中途異常） 每篇獨立 try/catch，失敗只跳過該篇，不中斷其他篇。不做 rollback。
R5 0 篇發布（所有分類全部失敗） 發布篇數為 0 時，透過 LINE 發送告警訊息：「0 篇發布，請手動檢查 RSS 來源。」
R6 RSS 來源暫時不可用 Promise.all + 10s timeout，單源失敗不阻塞其他來源，當天以剩餘來源繼續執行。
R7 ai-illustrate 失敗（文章無插圖） 可接受降級：文章仍以無插圖狀態發布，不跳過整篇。
R8 財經金融分類長期無文章 已於 v1.6（T-04）移除財經金融分類，不適用。
R9 Phase 2 週報：本週 D1 posts 為空 若當週 Phase 1 全部失敗（0 篇草稿），generateWeeklyInsights() 查無資料時直接跳過，LINE 推送告警「本週無新文章，insights 跳過」，不寫入空 insights row。

7.2 Cron 執行鎖詳細規格

Idempotency key 設計，防止 Cloudflare 對同一天重複觸發：

// index.ts - scheduled() 套件選項首行
export default {
  async scheduled(event, env, ctx) {
    const today = new Date().toISOString().slice(0, 10); // "2026-05-13"
    const lockKey = `lock:${today}`;
    const existing = await env.PIPELINE_DEDUP.get(lockKey);
    if (existing) {
      console.log(`[pipeline] already ran today (${today}), skipping.`);
      return;
    }
    // 寫入鎖（TTL 26 小時）
    await env.PIPELINE_DEDUP.put(lockKey, "1", { expirationTtl: 93600 });
    // ... 進入主流程
  }
};

8.4 LINE API 呼叫範例

// index.ts 內的工具函式
async function sendLine(env: Env, text: string): Promise<void> {
  await fetch('https://api.line.me/v2/bot/message/push', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${env.LINE_CHANNEL_ACCESS_TOKEN}`,
    },
    body: JSON.stringify({
      to: env.LINE_USER_ID,
      messages: [{ type: 'text', text }],
    }),
  });
}
長度限制： LINE 文字訊息最多 5,000 字元。完成摘要標題截取前 20 字為限。

十二、運維 FAQ

記錄初次部署實際遇到的問題與解法，供日後維護參考。

FAQ-01 — Worker 回傳 error code: 1101 / HTTP 500

症狀呼叫 /run endpoint 時，回傳 {"error":"error code: 1101"}，HTTP 500
根本原因原始 fetch() handler 用 await runPipeline() 直接等 pipeline 完成，但完整跑完 6 個分類需要約 14 分鐘 3 個分類（AI / 產業 / 資安）約 7 分鐘，遠超 Cloudflare Worker HTTP 請求的 30 秒限制，導致 runtime 強制終止並拋出例外。
修法改用 ctx.waitUntil()：fetch handler 立即回傳 {"status":"triggered"}，pipeline 繼續在背景執行不受 HTTP 超時影響。
// workers/auto-pipeline/index.ts — fetch handler 修法
async fetch(request: Request, env: Env, ctx: ExecutionContext) {
  // ... auth check ...
  ctx.waitUntil(
    runPipeline(env, { ignoreLock }).catch((error) => {
      console.error("[pipeline] uncaught error:", errorMessage(error));
    }),
  );
  return Response.json({ status: "triggered", ignoreLock });
}

FAQ-02 — Blog 文章點開顯示「找不到這篇文章」（404）

症狀首頁列表顯示 Pipeline 新產生的文章卡片，點進去卻顯示 404 頁面
根本原因 Aslugify() 函式的 regex \W 不匹配 CJK 字元，導致中文標題的 slug 保留了中文字（例如 dmtf-spdm標準納入fips-140-3-...）。Cloudflare Pages 將 URL 中的中文字 percent-encode 後傳入 [slug].ts，但 params.slug 取得的是 encoded 字串，直接查 DB 查不到。
根本原因 B即使解決 encoding 問題，slug 本身含中文也是非標準做法，SEO 較差且容易在各種環境中產生問題。

雙層修法：

Layer 1 — 修復現有文章（立即生效）：在 functions/api/posts/[slug].ts 加 decodeURIComponent，讓已存入 DB 的中文 slug 可正常查詢：

// functions/api/posts/[slug].ts
const slug = decodeURIComponent(params.slug as string); // 修改前為 params.slug as string

Layer 2 — 永久修復（新文章不再產生中文 slug）：修改 functions/api/admin/_auth.ts 的 slugify()，在處理前先移除 CJK 字元：

// functions/api/admin/_auth.ts
export function slugify(text: string): string {
  const ascii = text
    .toLowerCase()
    .trim()
    .replace(/[\u2E80-\u9FFF\uF900-\uFAFF\uFE30-\uFE4F\uFF00-\uFFEF]+/g, "-") // 移除 CJK
    .replace(/[\s\W-]+/g, "-")
    .replace(/^-+|-+$/g, "");
  if (!ascii) return Date.now().toString(36); // 純中文標題 fallback
  return ascii;
}
驗證：修復後，純 ASCII slug 文章（舊）和含中文 slug 文章（新 Layer 1 修復）都能正常開啟；之後 Pipeline 產生的新文章 slug 全為純 ASCII（例如 dmtf-spdm-fips-140-3-mp3uzu5o）。

13.1 背景與動機

PHASE 1 每天發佈 3 篇文章，一年累積約 1,000+ 篇。單篇閱讀無法看出趨勢——就像只看昨天的新聞無法判斷市場方向。PHASE 2 的目標是：

  • 每週一自動掃描 D1 posts 表（近 7 天的 published + draft），對 14 家追蹤企業分別計算命中篇數、提取關鍵事件。
  • 每家企業呼叫一次 GPT-5.5（最多 14 次/週），產出 direction + key_events + impact。
  • 分析結果寫入 D1 insights 表，每週只新增不覆蓋，形成可回溯的知識庫。
  • 前端「産業趨勢」頁（/trends）以時間軸呈現，可按公司 / 影響層級 / 時間範圍篩選。

13.3 D1 資料架構（0008_insights.sql）

CREATE TABLE IF NOT EXISTS insights (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  week_start    TEXT NOT NULL,           -- e.g. "2026-05-12"（週一日期）
  entity        TEXT NOT NULL,           -- e.g. "NVIDIA"
  entity_group  TEXT NOT NULL DEFAULT 'company',   -- e.g. "AI 研究" / "半導體" / "科技巨頭"
  hit_count     INTEGER NOT NULL DEFAULT 0,         -- 本週命中該企業的文章篇數
  direction     TEXT NOT NULL,           -- AI 產出的方向摘要，≤ 150 字
  key_events    TEXT,                    -- JSON array of notable titles
  impact        TEXT NOT NULL DEFAULT 'medium'
                  CHECK (impact IN ('high','medium','low')),
  source_ids    TEXT,                    -- JSON array of linked post IDs
  data_source   TEXT NOT NULL DEFAULT 'finmind',  -- 'FINMIND'（回填）| 'D1_POSTS'（週報 Cron）
  model_ver     TEXT,                    -- 分析時使用的 AI 模型，e.g. "gpt-5.5"
  prompt_ver    TEXT NOT NULL DEFAULT '1.0',
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_insights_week   ON insights(week_start);
CREATE INDEX IF NOT EXISTS idx_insights_entity ON insights(entity);
可回溯設計： 記錄每筆分析的 model_ver 與 prompt_ver，日後模型升級時仍能區分哪些分析由同一標準產出。
